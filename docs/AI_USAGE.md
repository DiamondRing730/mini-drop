# Mini-Drop AI 使用记录

本文记录 Mini-Drop 开发过程中对 AI 的使用，便于评审理解“人提出目标、AI 辅助实现、
人实际运行验收”的协作方式。根据导师答复，记录保留有代表性的用户指令和验证结果，
不要求提交冗长的完整对话。

## 使用范围

- 开发辅助：Claude Code、Codex，用于理解题目、拆解模块、生成与修改代码、定位故障、
  补测试和整理文档。
- 产品能力：Mini-Drop 页面中的“离线归因 / DeepSeek 归因”。外部模型必须由用户显式
  选择并配置 API Key；默认离线，不会静默调用第三方服务。
- AI 没有替代真实性验收。核心结果均通过 WSL2、Docker、真实 py-spy/perf/eBPF 采集、
  单元测试和端到端测试核验。

## 代表性协作记录

| 阶段 | 用户指令（摘录或等义整理） | AI 辅助内容 | 人工/工具验收 |
|---|---|---|---|
| 环境准备 | “WSL2 行吗”“怎么控制两个不同版本的系统” | 选择 Ubuntu 22.04，排查 apt、perf、bpftrace、py-spy、Docker Desktop 集成 | perf 采样成功、bpftrace tracepoint 成功、py-spy SVG 成功、hello-world 成功 |
| 核心闭环 | “分析一下项目”“现在完成了什么” | 实现 Web→Server→Agent→采集器→Analyzer→结果展示 | 三条 E2E：正常、坏 PID、Agent 离线恢复 |
| 火焰图 | “为什么没有嵌到网页里”“修复你说的 bug” | 修复 SVG 响应/内嵌显示，统一页面展示 | 浏览器中直接显示真实火焰图 |
| 扩展采集 | “继续完成” | 增加 eBPF 延迟采集、py-spy Continuous Profiling、窗口回放 | syscall 分布与持续切片实跑 |
| 智能归因 | “去掉 Claude 调用，页面选择离线归因还是 DS 归因” | 移除 Claude 路径，显式选择离线或 DeepSeek，增加独立证据校验 | 离线归因无需 Key；DeepSeek 无 Key 明确报错 |
| 优化闭环 | “按你说的做可验证的性能优化闭环” | 前后任务比较、函数归一化、差分火焰图、独立复算报告 | fib 热点由 74.22% 降至 0%，校验 5/5 通过 |
| 演示场景 | “多创建几个演示任务，分别运行，同一程序两个版本” | 拆分 CPU 优化前/后、数值、I/O 场景及独立 Make 命令 | 每个命令只启动并采集一个场景 |
| 任务控制 | “Continuous 主动停止与续建” | 增加 STOPPED 状态、心跳停止控制、续建新任务并保留旧时间轴 | API、状态机、页面操作与实跑测试 |
| 目标发现 | “页面直接选择容器和 PID” | Agent 只读查询 Docker socket，回报容器与宿主 PID，页面联动选择 | 容器、进程、PID 可视化选择；手动 PID 仍可用 |

## AI 参与边界

AI 主要承担高频代码生成、重构建议和故障分析。以下关键决策与验证由用户主导：

1. 明确以“完整复刻、真实跑通、演示效果”为目标，而不是只做静态界面。
2. 选择 WSL2 Ubuntu 22.04 + Docker Desktop 作为可复现演示环境。
3. 亲自运行 perf、bpftrace、py-spy、Docker、浏览器页面并反馈真实输出。
4. 要求归因结论必须有采样证据和复算校验，优化结果必须进行前后对比。
5. API Key、凭据、密码和完整私密对话不进入仓库。

## 复现与校验

```bash
make unit
make e2e
make demo-before
make demo-after
```

评审时可从页面完成：选择容器/PID → 创建任务 → 查看火焰图与热点 → 运行归因 →
选择优化前任务进行对比；持续任务还可主动停止，并通过“续建采样任务”创建新的连续会话。
